功能测试
2024/11/17

* 该模块最重要的其实是写好项目功能的测试用例，越全越好

服务器（环境）组成：操作系统+数据库+web应用程序+项目代码

B/S架构：客户端使电脑浏览器向web发出请求，web向数据库发出请求

，数据库接受请求发出响应到web，web响应客户端浏览器

基本项目环境：LNMP(Linux，nginx，MySQL，php项目)和WAMP（Windows+apache+MySQL+php项目）

一般有几套环境？由于开发，测试，运维，用户每一环节的环境都不一样，所以基本会有4套，测试环境，开发环境，预生产环境（上生产环境前测试的最终测试），生产环境（用户使用的环境）

什么是功能测试？

对产品的各项功能验证，根据测试用例逐项测试检查产品能否达到用户要求的功能

根据产品的需求文档对功能进行测试，功能测试也叫做黑盒测试不用知道程序代码怎么写的只需要知道功能的逻辑是什么样的。

功能测试占据大部分的测试比重。

详细写测试用例的方法？

1. *等价类测试（穷举场景，只关注类型）*

(1) 等价类划分步骤：（有效等价类：满足需求的数据集合；无效等价类：不满足需求的数据集合）1.明确需求

(2) 确定有效和无效等价类

(3) 3.提取数据编写测试用例（取有效和无效其一即可）。【案例：去6~10位自然数QQ，有效：6<=x<=10位自然数;无效：x<6 and x>10位自然数】

(4) {企业中测试主要难点为三类，长度，类型，规则，看excel等价类作业}

(5) 针对场景：需要大量数据测试输入但是没法穷举的地方：

① 输入框

② 下拉列表

③ 单选框 复选框

④ 典型代表：页面的输入框类测试

2. *边界值分析法（限定边界规则场景，只考虑长度问题，不考虑类型）*

(1) 边界范围节点：选取正好等于，刚好大于、刚好小于边界的值作为测试数据

① 上点：边界上的点（正好等于）

② 离点：距离上点最近的点（【一定有两个】刚好大于、刚好小于）

③ 内点：范围内的点（区间范围内的数据）

(2) 边界值法设计步骤：

① 明确需求

② 确定有效和无效等价类(用等价类做类型校验)

③ 确定边界范围值

④ 提取数据编写测试用例

(3) 边界优化：

① 上点：必选（不考虑开区间）

② 内点：必选（选择中间范围，验证范围的连续性）

③ 离点：开内闭外（开区间选内部离点，闭区间选外部离点）

(4) 针对场景：等价类基础上针对有边界范围的测试数据输入的地方

① 常见描述：大小，尺寸，重量，最大，最小，至多，至少

② 典型代表：有边界范围的输入框类测试

3. *判定表法（多条件依赖关系场景：要求条件间存在依赖关系，例如省份下属的市区，其他省份没有）*

(1) 定义：是一种以表格形式表达多条件逻辑判断的工具

(2) 组成：

① 条件桩：列出问题所有条件，无关次序

② 动作桩：列出问题可能采取的操作，无关次序约束

③ 条件项：列出条件对应取值，所有可能情况下的真假值

④ 动作项：列出条件项的、各种取值情况下应该采取的动作结果

(3) 规则：

① 判定表中贯穿条件项和动作项的一列就是一条规则

② 假设有n个条件，每个条件的取值有两个（0，1），去组合有2^n种。

4. *场景法（项目业务场景：测试业务的核心功能能否正常实现，例如电商，搜索，添加购物车，下单，支付，售后这一条龙业务能否走通。）*

(1) 流程图（一般由产品经理提供，visio绘画）

(2) 业务就是流程图使用“->”描述完整功能路径

(3) 冒烟测试：批量开始测试之前，执行业务正向用例

① 提测之前开始执行，验证软件是否具备可测性，避免人力成本浪费。

5. *错误推荐法*

(1) 定义：通过经验推测系统可能出现的问题

(2) 思想：根据经验列举出可能出现问题的清单，根据清单分析问题可能原因，推测发现缺陷

(3) 场景：

① 时间紧任务量大时，根据之前项目类似经验找出易出错的模块重点测试

② 时间宽裕通过该方法列出之前出问题较多的模块再次测试

测试缺陷如何管理？

将执行结果与预期结果不符合的测试用例收集在excel内。

\1. 软件缺陷判定标准

(1) 缺陷介绍

① 软件使用过程种出现的任何问题都叫软件的缺陷，简称bug。

② 缺陷的评价标准：

\1) 少功能

\2) 功能错误

\3) 多功能

\4) 隐形功能错误（除测试用例外该关注的标准）

\5) 不易使用

③ 缺陷产生的原因：

a. 需求阶段：需求描述不易理解，有歧义、错误等（产品经理会提出需求）

\1) 设计阶段：设计文档存在错误或者缺陷（架构师会细化到每个字段的定义）

\2) 编码阶段：代码实现出现问题。（前后端开发使用代码实现）

\3) 运行阶段：软硬件系统本身故障导致软件缺陷（系统不兼容，硬件不满足条件）

④ 软件缺陷的生命周期：

\1) 需求规格说明书（缺陷）–设计（缺陷）–编码（缺陷）–测试–故障分类–故障隔离–故障解决（缺陷）

⑤ 软件缺陷的核心内容（重要）

\1) 缺陷的标题（描述缺陷的核心问题，最难写）

\2) 缺陷的预置条件（缺陷产生的前提）

\3) 缺陷的复现步骤（复现缺陷的步骤）

\4) 缺陷的预期结果（希望得到的结果）

\5) 缺陷的实际结果（实际得到的结果）

\6) 缺陷的必要附件（图片、日志等信息（证据））

⑥ 缺陷提交要素（重要）

\1) 缺陷的报告编号（缺陷的唯一标志）

\2) 却显得严重程度（s1:主功能，s2：次功能，s3：易用性，界面，s4：建议性问题）

\3) 缺陷优先级（priority 0：24小时内解决；priority 1：发布前必须修复；priority 2：可以在下一个版本中修复）

\4) Bug类型（代码错误，兼容问题，设计缺陷，性能问题）

\5) 缺陷状态（new:新建；open:打开；closed：关闭；postponed：缺陷状态）

⑦ 软件缺陷类型：

\1) 功能错误（后端出错）

\2) 界面错误（前端出错）

\3) 兼容性错误（前端出错）

\4) 数据错误

\5) 易用性错误（2、3一般会引起该错误）

\6) 改进建议

\7) 架构错误

\2. 缺陷编写(详见excel-缺陷报告示例)

一、提交注意事项

\1. 可重现（缺陷可以重现）

\2. 唯一性（一个缺陷报一个问题）

\3. 规范性（符合公司或项目要求）

二、编写注意事项

\1. 准确（描述的信息是正确的）

\2. 简洁易懂（描述简单容易理解）

\3. 具体（有细节且是真实特定的）

\4. 次序清晰（描述缺陷过程有条件，有先后顺序）

\3. 缺陷管理工具

Welcome To My Blog!
2024/11/17

个人blog创建方法

安装nodejs和git

首先进入https://nodejs.org/按照电脑系统下载对应.msi文件，双击文件安装，选择安装路径的时候建议在空间较大的盘内新建一个命名为nodejs的文件夹专门用来安装,右键点击该文件，点击安全，点击编辑，点击权限，全选 ，win+D输入cmd输入nvm -v 检查有无版本号，有版本号即代表安装成功，安装成功后输入npm config set registry https://registry.npmmirror.com 切换下载镜像源，国内没有挂梯子下载会很慢，换镜像源下载速度会提升许多，cmd内输入npm config get registry检查安装的镜像源是否成功

在nodejs文件夹内新建node_cache和node_global两个新文件夹，在路径处全选输入cmd回车，输入npm config set prefix “盘名:\nodejs\node_global” npm config set cache “盘名:\nodejs\node_cache”

用户变量下的Path点击进去将npm修改成 盘名:\nodejs\node_global 点击确定，在系统变量下新建NODE_PATH,变量值为 盘名:\nodejs\node_global\moudles，在系统变量的Path新建 盘名:\nodejs\node_global,这样后续下载的模块会自动导入指定路径

在cmd内输入npm install express -g，下载成功 盘名:\nodejs\node_global\moudles文件下会新增一个express文件夹

安装git工具，提前创建github账号，没有梯子就输入kkgithub.com进入镜像

git下载网址：https://git-scm.com/ 选择最新安装包，下载好后双击下一步，如上提前新建好一个名为git的文件夹，安装在该文件夹内，第五步有“Don’t create a start menu folder”要打勾，然后下一步就行。

cmd输入git -v查看到版本信息即是安装成功，在一个较大盘内新建名为hexo的文件夹，git bash here打开窗口
输入npm install -g hexo-cli安装hexo框架，输入hexo -v出现版本信息说明安装成功
在新建的文件夹内右键git bash here输入hexo init ”项目名“ 初始化项目，项目名尽量起的与后续工作相关

cd 项目名 进入这个文件夹 npm install 暂时只需要关注初始化成功后出现的source文件内_post文件内的博客md文件（存文章），theme文件夹（存博客主题），_config.yml文件（博客配置信息）
theme文件主要是用来下载hexo主题的，可以去hexo官网寻找喜欢的主题git clone代码到这个文件夹
最后把_config.yml文件内的theme和最下面.deploy文件信息配置好theme这里的名字和下载的名字必须一模一样，_post文件夹里面的两个文件名也是必须一模一样，建议复制粘贴，不然hexo d的时候会出error

创建好github账号就创建一个个人仓库，起名为用户名.github.io方便记，将c盘内.ssh文件生成的ssh添加到github，再将hexo部署到github，git bash here 输入npm install hexo-deployer-git –save

hexo部署方法为打开_congif.yml文件拉到最下面.deploy输入:
type: git
repo: https://github.com/用户名/用户名.github.io.git //和上面创建的仓库名一致
branch: master //或者main，看你提交文件左上角是哪一个

搞定好就在git窗口里hexo cl && hexo g &&hexo s 在浏览器输入localhost:4000查看本地博客页面情况，最后hexo g && hexo d部署，git pull 远程同步一下,hexo g && hexo d 的时候会因为git代理问题需要多提交几次才会成功，提交成功后直接在浏览器输入https://用户名.github.io观察博客情况

每次更新都会觉得麻烦的话可以新建个deploy.sh文件，把以下代码放进去，每次更新完就在bash里面执行 sh deploy.sh

#!/bin/sh
hexo clean
hexo generate
hexo deploy
